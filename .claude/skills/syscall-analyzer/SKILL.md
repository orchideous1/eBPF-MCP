# 系统调用分析技能

## 与 nfs-sre-expert 的关系

- **被委托时**：`nfs-sre-expert` 已制定跨层观测计划，你负责 **Syscall 层**的深入分析，为上层（NFS / RPC）提供进程行为上下文。
- **独立调用时**：你需自行决定采集策略、完成观测闭环并输出报告。

---

## 核心分析能力

| 能力 | 说明 |
|------|------|
| **频率分析** | 识别进程最活跃的系统调用，发现异常调用模式 |
| **延迟分析** | 定位高延迟系统调用，识别 P95/P99 长尾 |
| **进程画像** | 按 `comm` / `pid` 聚合，区分不同进程的系统调用行为 |
| **错误追踪** | 通过 `ret < 0` 发现系统调用层面的失败 |

---

## 数据语义

系统调用数据存储在 `sys_call_trace` 表中：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| pid | UBIGINT | 进程ID (TGID << 32 \| PID) |
| syscall_id | UINTEGER | 系统调用号（数字形式） |
| ret | BIGINT | 系统调用返回值 |
| duration | UBIGINT | 调用延迟（纳秒） |
| enter_time_stamp | UBIGINT | 进入时间戳（纳秒，自系统启动） |
| comm | VARCHAR | 进程名称（最多16字符） |

**关键语义**：
- `syscall_id` 是数字，分析阶段先在 SQL 中保持数字形式，报告阶段再映射为名称。
- `ret < 0` 表示系统调用返回错误。
- `duration` 和 `enter_time_stamp` 均为纳秒级，报告时通常转换为毫秒或秒。

## 独立观测闭环

### 1. 加载探针
通过 `system_observe_control` 加载 `sys_call_trace` 探针。按需使用 `probe_customize` 设置 `filter_pid`、`filter_comm` 或 `filter_syscall_id`。

**过滤策略**：
- 全量采集：适用于未知问题排查，但性能开销大。
- 按 `pid` / `comm` 过滤：适用于已知目标进程的分析。
- 按 `syscall_id` 过滤：适用于关注特定系统调用（如只关注文件操作）。

### 2. 采集与卸载
| 分析类型 | 建议采集时间 |
|----------|-------------|
| 简单分析 | 10-30 秒 |
| 性能分析 | 1-5 分钟 |
| 异常排查 | 直到问题复现 |

采集完成后**必须卸载探针**，获取最新 DuckDB 文件路径并验证数据已写入。

### 3. 分析思路

**性能问题**：优先关注 `duration` 的 P50/P95/P99 分布，定位高延迟调用。
**行为分析**：优先统计 `syscall_id` 频率和 `comm` 分布，理解进程在做什么。
**异常排查**：优先筛选 `ret < 0` 的记录，结合错误码和调用上下文分析。

### 4. 系统调用号映射

分析阶段保持 `syscall_id` 为数字，报告阶段参考 `resource/unistd.h` 映射为可读名称。常用编号速查：

| 编号 | 名称 | 功能 |
|------|------|------|
| 0 | read | 从文件描述符读取 |
| 1 | write | 向文件描述符写入 |
| 2 | open | 打开文件 |
| 3 | close | 关闭文件描述符 |
| 257 | openat | 相对于目录打开文件 |

## 报告结构

1. **执行摘要** - 关键发现和结论（3-5 条要点）
2. **采集概况** - 数据量、时间范围、进程覆盖、采样时长
3. **频率分析** - 热点系统调用、分布统计、调用模式
4. **延迟分析** - 延迟分布、P95/P99 长尾、瓶颈识别
5. **进程分析** - 各进程系统调用行为、资源消耗排行
6. **异常发现** - 错误返回、高延迟事件、异常模式
7. **优化建议** - 基于分析数据的具体建议
