# Go 项目开发规范

本规范用于约束本仓库内 Go 项目的代码风格、工程结构、环境管理与质量门禁。

## 第一部分：核心开发原则

### 基础设计原则
- 可读性优先：代码应简洁、可维护，优先选择标准库与清晰命名。
- DRY：通过函数、包、接口抽象复用逻辑，避免重复实现。
- 高内聚低耦合：按领域职责组织 package，减少跨层依赖和循环依赖。

### DDD + TDD 融合方法
- 领域驱动设计：围绕 domain model 组织代码，先定义业务边界再落地实现。
- 测试驱动开发：优先写失败测试（Red），再写最小实现（Green），最后重构（Refactor）。
- 质量保障：结合 `go test`、`go vet`、`golangci-lint`（若项目已配置）进行静态和行为校验。

## 第二部分：具体执行指令

### 沟通与语言规范
- 默认语言：交流与解释使用简体中文。
- 代码实体：变量名、函数名、类型名、包名、接口名保持英文。
- 注释规范：注释与文档可使用中文，但应精确描述“是什么、为什么”，避免赘述“做了什么”。
- 所有类和函数的抽象在写之前都需要经过用户的详细讨论，等和用户统一方案之后再动手。

### Go 代码风格与惯例
- 必须使用 `gofmt` 与 `goimports`，禁止手动维持格式和导入顺序。
- 错误处理必须显式处理，不忽略 `error` 返回值。
- 包装错误使用 `fmt.Errorf("...: %w", err)`，保留错误链。
- 禁止在库代码中随意 `panic`；仅在不可恢复、明确约定场景使用。
- `context.Context` 作为首个参数传递到跨边界调用（I/O、RPC、DB、并发任务）。
- 接口应小而专，优先在消费方定义接口，不为“未来可能”预抽象。
- 函数职责单一，控制圈复杂度；超过可读阈值时拆分函数。
- 并发代码必须具备取消机制与资源回收路径，避免 goroutine 泄漏。
- 时刻保持代码简结，减少不必要的封装，如：func OuterFunc(){return innerFunc()}

### 环境与依赖管理
- Go 版本基线：与 `go.mod` 保持一致（当前为 Go 1.23.0）。
- 使用 Go Modules 管理依赖，禁止 GOPATH 模式开发。
- 每次变更依赖后运行 `go mod tidy`，保证 `go.mod` 与 `go.sum` 干净一致。
- 优先使用标准库；引入第三方依赖前，评估维护活跃度、许可证和安全风险。

### 测试与质量门禁
- 每当你要写测试的时候，都先与用户确认具体测试项。
- 先写集成测试，当集成测试无法定位根因时再写单元测试，保持测试的简结和高效。
- 核心业务逻辑覆盖率目标不低于 90%。
- 提交前至少执行：
  - `go test ./...`
  - `go test -race ./...`
  - `go vet ./...`
  - `golangci-lint run`（若仓库已配置）

### 可观测性与错误治理
- 日志应包含可定位上下文（request id、资源标识、关键参数摘要）。
- 不记录敏感信息（token、密码、密钥、完整隐私数据）。
- 对外返回错误信息时避免暴露内部实现细节。

## 常用命令

```bash
# 编译eBPF探针
# 进入探针目录
go generate

# 依赖整理
go mod tidy

# 单元测试
go test ./...

# 竞态检测
go test -race ./...

# 静态检查
go vet ./...

# 代码格式化
gofmt -w .
goimports -w .

# 质量检查（若项目已安装配置）
golangci-lint run
```

## 强制要求

1. 所有新增和修改的 Go 代码必须可通过 `gofmt` 与 `go test`。
2. 所有公开函数、方法、类型必须具备清晰文档注释。
3. 所有错误必须显式处理，不得静默吞错。
4. 核心业务逻辑必须有测试，覆盖关键分支与失败路径。
5. 并发实现必须具备超时、取消和回收机制。
6. 依赖变更必须同步更新 `go.mod` 与 `go.sum`。
7. 禁止提交包含敏感信息的代码与配置。
